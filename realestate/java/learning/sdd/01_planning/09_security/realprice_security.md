# Security: RealField 실거래 수집·조회 보안 요구 (realprice_security)

> SDD 1단계(01_planning) 정본. `00_sources/02_requirements/realfield-부동산실거래.md`의
> 보안요구사항(SECR)을 위협·통제 모델로 정제한다. 기능 수용기준은
> `01_planning/01_feature/realprice_ingest.md`가 정본이다.

---

## 1. 보안 목표

- 외부 API 인증키(`serviceKey`, 환경변수명 `MOLIT_SERVICE_KEY`)를 코드·이미지·설정 파일에 평문 노출하지 않는다.
- 인증키를 포함한 어떤 값도 로그·트레이스·에러 응답에 남기지 않는다.
- 본 시스템이 다루는 실거래 데이터에 개인 식별정보가 없음을 지속적으로 보증한다.
- 인증키 만료·교체가 서비스 중단 없이 반영되게 한다.

## 2. 위협·통제 모델

| --- | --- | --- | --- |

| SEC-002 | 인증키·엔드포인트·회복력 정책이 각 서비스에 분산되어 일관성 없이 관리됨 | 인증키·엔드포인트·resilience4j 정책을 Config Server로 외부화하고, 비밀값은 환경변수 참조로만 처리한다. | SECR-002 |
| SEC-003 | 로그·트레이스·에러 응답에 인증키가 노출됨 | 수집 서비스의 HTTP 클라이언트 로깅·예외 메시지·분산 트레이싱 태그에서 `serviceKey` 쿼리 파라미터를 마스킹한다. 요청 URL을 그대로 로깅하지 않는다. | SECR-003 |
| SEC-004 | 적재 데이터에 개인 식별정보가 포함되어 개인정보 규제를 위반함 | 원천 API는 거래 당사자 식별정보를 포함하지 않는다(동(棟) 정보는 등기 완료 건에 한해 비식별 수준으로만 공개, 매도자/매수자 구분은 비식별 범주값). 적재 스키마에 성명·연락처 등 식별자를 추가하지 않는다. | SECR-004, DAR-004(cdealType 도메인), 데이터명세서 §2.2 |
| SEC-005 | 인증키 만료·교체 시 서비스 중단 또는 반영 지연 | 인증키를 Config Server 값으로 외부화해 무중단 설정 갱신(리프레시)으로 반영 가능하게 한다. | SECR-005 |

## 3. 통제 상세

### 3.1 인증키 비노출·외부화 (SEC-001, SEC-002)

- 저장 위치: 인증키는 배포 환경변수(`MOLIT_SERVICE_KEY`)로만 주입한다. `.env`, 자격증명 파일, 컨테이너
  이미지 레이어에 값이 남지 않아야 한다.
- Config Server: `config-server/src/main/resources/config/ingestion-service.yml`은 `service-key: ${MOLIT_SERVICE_KEY:}`
  형태의 참조만 보관한다(2026-07-02 기준 코드 확인, 실값 없음 — 현재 상태는 SECR-001을 충족).
- 리포지토리 정책: 실 인증키, 브라우저 상태, 특정 서비스/실환경 자격증명을 커밋하지 않는다(`.claude/CLAUDE.md` Working Rules).

### 3.2 로그 마스킹 (SEC-003)

- 수집 서비스가 외부 API 호출 URL·쿼리 파라미터를 로깅할 때 `serviceKey` 값을 마스킹(`***`)한다.
- 에러 응답·예외 스택트레이스에 원본 요청 URL을 그대로 포함하지 않는다.
- 검증 방법: 재시도·실패 케이스 로그를 캡처해 인증키 평문이 출력되지 않음을 검사한다.

### 3.3 개인정보 미보유 확인 (SEC-004)

- 데이터 사전(`00_sources/03_data_spec/realprice_data_spec.md` §1~2) 기준으로 원천 항목 전체를 검토했으며,
  성명·연락처·주민등록번호 등 식별자 항목은 없다.
- `slerGbn`/`buyerGbn`(매도자/매수자 구분)은 개인/법인/공공기관/기타의 비식별 범주값이다.
- `aptDong`(동 정보)은 등기 완료 건에 한해서만 공개되며 개별 소유자를 특정하지 않는다.
- 표준 스키마(AptTransaction) 설계 시 위 항목 외 식별정보 필드를 추가하지 않는다(향후 `01_planning/04_data`에서
  스키마 확정 시 본 절을 검토 기준으로 재확인한다).

### 3.4 키 만료·교체 무중단 반영 (SEC-005)

- 인증키는 Config Server 값으로 관리하여, 키 교체 시 서비스 재배포 없이 설정 갱신으로 반영 가능한 구조를
  목표로 한다(구체적인 리프레시 메커니즘은 `01_planning/07_integration`에서 확정, 후속 작성 대상).
- 키 반영 지연(최대 1영업일, `00_sources/01_apis/molit_apt_trade_api.md` §7)을 감안해, 신규 키 발급은
  기존 키 만료 이전에 선제 신청한다(운영 절차, `05_operate/01_runbooks` 후속 반영 대상).

## 5. 범위 밖

- 게이트웨이·서비스 간 내부 인증(mTLS, 서비스 계정 등)은 본 원천 문서(SECR)에 명시되지 않아 범위 밖이다.
  플랫폼 경계(T4) 설계 시 별도 검토가 필요하면 `01_planning/09_security`에 후속 절로 추가한다.
